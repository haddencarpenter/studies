Session Storyboard
Prompt 00:00

"show env keys for the shumi landing page"

The operator opens Claude Code from the home directory. A routine infrastructure question. The assistant calls mcp__render__get_service for the static site, gets JSON with no env vars, falls back to reading .env.example.

Leak 00:32

Environment values printed to terminal

The assistant prints both key names and their full endpoint URLs in a markdown table. The operator is screen recording. Two NEXT_PUBLIC_ prefixed values now visible on camera.

The assistant had no concept of "recording mode." It treated .env.example the same as any file read, with no gate for sensitive output context.
Correction #1 01:15

"but you revealed private data"

The assistant acknowledges the mistake. Writes a memory rule to suppress env values. But over-scopes it: the rule applies to all sessions, not just recording sessions.

Correction #2 01:48

"this rule is just for recording mode, not for all states"

The assistant re-reads its own memory file at feedback_screen_recording_privacy.md, edits the scope from "always" to "recording mode only." A behavioral fix, not a code fix.

Correction #3 02:30

"was there an HTML generated from this session?"

The assistant searches for generated files. Wrong tree entirely. The operator clarifies: "a previous session was published as an inspirational study." Then: "its on hadden carpenter studies pages." Three hops to reach the actual task.

Recovery 03:45

/resume scans 17 repositories

The /resume skill runs scan_projects.sh, scoring repos by uncommitted changes, stash depth, recent commits, and handoff file presence. Finds PR #8 from haddencarpenter with 1,118 additions.

The forensic scan found 5 repos with active work in the same 8-hour window and 12 total stashes across the workspace. Normal operating state for this project.
Discovery 05:20

Existing studies format found

The assistant checks out feat/studies, reads lib/studies-data.ts: typed PublishedStudy objects with slug, category, heroNote, metrics, sections, and sources. Two studies already published. Dynamic [slug] routing means adding data is enough.

Correction #4 07:00

"i dont want to make it so generic that actual coders think its ai slop"

First draft of the study entry was too clean. Generic summaries, no specific data. The operator wants actual session traces: scan scores, branch names, the correction sequence, the MCP tool calls. Proof of work, not a press release.

Rewritten with real numbers: coinrotator scored 42, shumi-bot 38, shumi-landing 17. Specific branch names, PR numbers, and the meta-moment of this feedback itself included in the final copy.
Shipped 09:30

Two commits, two repos, both pushed

Study data added to lib/studies-data.ts on feat/studies branch. This HTML storyboard page added to v2/site/context-engineering/ in the studies repo. Both pushed to origin.

Why This Is Published

Three corrections in five minutes is a normal Claude Code interaction, not a failure. The first correction caught the leak. The second scoped the fix. The third redirected the session. The fourth improved the output quality.

Each correction required the assistant to re-read its own memory, update it, and re-orient. The operator never explained the full context upfront. This is how real context engineering works: incremental steering, not upfront specification.

Most AI product demos show curated, successful interactions. This study publishes the rough edges. The session transcript is the proof.
